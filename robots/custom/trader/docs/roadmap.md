# Trader 演进路线图

本文档和 `docs/baseline.md` 配套使用。

- `baseline.md` 回答的是：当前已经落地了什么；
- `roadmap.md` 回答的是：接下来应该先做什么、后做什么、哪些事情现在不要急着做。

这里的路线图以“当前 Trader 机器人”作为出发点，不预设未来一定重构成另一套架构。优先顺序以可验证、可收敛、可控风险为原则。

补充约束：

- Trader 后续默认按“交易 agent”方向持续演进；
- 所有阶段都建议围绕 `观察 -> 理解 -> 筛选 -> 触发 -> 计划 -> 执行 -> 复盘` 这条闭环补能力；
- 不再把它当成单一策略脚本来规划。

## 1. 总体原则

Trader 后续演进遵循四个原则：

1. 先把现有 MVP 做清楚，再谈生产级扩展；
2. 先补可验证闭环，再补高复杂度策略；
3. 先保证审计、状态和回放，再增加外部接口调用；
4. 先把 stub / paper 跑稳定，再接真实券商。

如果从 agent 视角重写，上面四条也可以理解为：

1. 先补观察、筛选和记录能力，再补复杂执行；
2. 先让 agent 的中间产物可解释，再追求自动化；
3. 先让每一步都可回放，再增加外部依赖；
4. 先做受控决策支持，再做高风险实盘动作。

## 2. P0：当前阶段必须完成

P0 的目标是把当前 Trader 从“能跑”提升到“能稳定验证、能复盘、能持续演进”，并先形成最小可用的交易 agent 骨架。

### 2.1 运行结果固化与回放

目标：

- 降低同花顺接口重复消耗；
- 支持事后分析；
- 支持策略迭代时的回放验证；
- 为 agent 的观察、触发、计划产物提供统一留痕。

建议工作：

1. 为每次 `run_once` 生成独立的运行批次 ID；
2. 把本轮使用到的行情快照、策略输入、风控结果、订单计划保存到批次目录；
3. 增加“按历史批次回放”的能力；
4. Hub 中可以按批次查看，而不是只看滚动最新状态。

建议产物：

- `runtime/runs/<run_id>/summary.json`
- `runtime/runs/<run_id>/market.json`
- `runtime/runs/<run_id>/signals.json`
- `runtime/runs/<run_id>/orders.json`

### 2.2 先补 agent 的中间产物模型

目标：

- 不让 Trader 只有最终 signal / order，而是先形成 agent 可复用的中间产物。

建议工作：

1. 明确市场观察输出；
2. 明确候选池输出；
3. 明确 trigger 输出；
4. 明确订单计划输出；
5. 明确复盘记录输出。

建议产物：

- `market_observation`
- `watchlist_candidates`
- `entry_trigger`
- `exit_trigger`
- `order_plan`
- `run_review`

完成标准：

- 单看运行记录，不只能看到“最后买卖了什么”，还要能看到“为什么关注、为什么触发、为什么计划这样执行”。

### 2.3 `auction_wave` 卖出规则显式化

目标：

- 把现在“写在代码里、说在文档里”的卖出规则，明确变成统一可审查的策略定义。

建议工作：

1. 把 `09:29`、`0%`、`3%`、`14:30`、`涨停阈值` 这些关键阈值整理为显式配置；
2. 输出每个判断分支的决策原因；
3. 为“卖一半 / 全卖 / 持有”生成统一的动作枚举；
4. 把“已经卖过一半”这类中间状态稳定写入 `state.json`。

完成标准：

- 单看运行记录，就能解释为什么发出这个动作；
- 不需要再去猜代码分支。

补充说明：

- 这一阶段优先补卖出主路径，不以补完整买入逻辑为目标。

### 2.4 策略记录模型统一

目标：

- 让 Hub 展示、日志落盘、后续分析都使用同一套记录结构。

建议工作：

1. 统一 signal / order / risk / snapshot 的字段模型；
2. 每条记录带上 `strategy_id`、`run_id`、`symbol`、`ts`、`decision_reason`；
3. 把当前 JSONL 中不稳定字段逐步收口。

完成标准：

- 同一条记录在 CLI、运行文件、Hub 页面里的语义一致。

### 2.5 Trader 工作台和运行目录对齐

目标：

- 当前前端已经能看，但“运行批次”和“滚动摘要”还没有完全对齐。

建议工作：

1. 首页增加最近运行批次列表；
2. 单策略页支持按批次筛选记录；
3. 记录详情页展示 `run_id`、数据来源、决策阶段；
4. 前端尽量从“运行批次”而不是“散乱的 recent_signals / recent_orders”出发组织页面。

## 3. P1：下一阶段重点建设

P1 的目标是把 Trader 从“可验证 MVP”推进到“可控的准生产交易 agent”。

### 3.1 市场层与标的层建设

目标：

- 让 Trader 具备真正的“观察与筛选”能力，而不是只对预设标的执行规则。

建议工作：

1. 板块 / 题材热度排序；
2. 市场环境判断；
3. 候选池筛选；
4. 形态位置判断；
5. 压力位评估；
6. A/B/C 级观察池分层。

完成标准：

- 每次运行都能输出“今天看什么、为什么看、哪些票进入观察池”。

### 3.2 买入触发与盘口确认

目标：

- 建立真正决定效果上限的买点能力。

建议工作：

1. 买入 trigger 模型；
2. 盘口结构信号模型；
3. 冲板 / 回封 / 卖单消失等行为建模；
4. 试仓 / 确认 / 追加的订单计划模型。

前提条件：

- 先确认能拿到哪些稳定盘口数据；
- 如果拿不到完整盘口，就要明确近似方案，而不是假装具备完整盘口能力。

### 3.3 风控体系增强

目标：

- 从“单条策略风控”扩展到“账户层和 agent 层风控”。

建议工作：

1. 单策略最大仓位；
2. 单日最大亏损；
3. 单日最大成交次数；
4. 账户总暴露限制；
5. 禁止重复下单和重复卖出保护；
6. 外部接口异常时降级为只观察、不下单。

### 3.4 测试体系补齐

目标：

- 当前 Trader 更偏手工验证，后续要逐步补自动化。

建议工作：

1. 市场层 / 标的层判断器测试；
2. 买入与卖出 trigger 测试；
3. 风控规则单元测试；
4. broker 载荷测试；
5. 运行批次输出集成测试；
6. Hub 读取 Trader 数据的接口测试。

### 3.5 真实东方财富适配器

目标：

- 从 `eastmoney_stub` 走向真实下单，但必须保留强约束。

建议工作：

1. 新建真实 `eastmoney` broker；
2. 引入显式开关，默认关闭；
3. 增加执行前检查：
   - 环境确认；
   - 账户确认；
   - 标的确认；
   - 数量确认；
   - 风控确认；
4. 所有真实下单动作都要写完整审计日志。

前提条件：

- P0 的记录模型和回放机制已经完成；
- 不能在没有可回放审计的情况下接真实券商；
- 不能在没有市场层、标的层、触发层输出的情况下直接跨到实盘。

## 4. P2：生产化与平台化

P2 的目标是把 Trader 从单机器人 MVP，推进到可接入更大机器人体系的交易 agent 模块。

### 4.1 多机器人 / 多账户

建议工作：

1. 一个 Trader 管理多个账户；
2. 一个 Hub 管理多个交易机器人实例；
3. 支持按账户、按策略、按市场隔离状态和审计。

### 4.2 机器人化标准接口

建议工作：

1. 给 Trader 补统一 robot manifest；
2. 把启动、状态、配置、记录导出成统一机器人协议；
3. 让 `openclaw` / `nanobot` / `trader` 在 Hub 里都能走相同的管理接口。

### 4.3 认证与授权闭环

建议工作：

1. Hub 鉴权接入 UCAN / web3-bs 能力；
2. 区分查看权限、配置权限、执行权限、实盘权限；
3. 所有高风险动作增加显式授权与审计。

### 4.4 事件与告警

建议工作：

1. 接口异常告警；
2. 风控触发告警；
3. 下单失败告警；
4. 服务停摆告警；
5. 关键动作消息通知。

## 5. 当前不建议优先做的事

下面这些事情现在可以先明确“不急着做”：

1. 一开始就把 Trader 抽象成过度复杂的通用量化框架；
2. 一开始就接很多券商；
3. 在没有审计回放前直接推真实实盘；
4. 在市场层、标的层、触发层还不清楚时过早做大规模参数优化；
5. 在同花顺额度有限的情况下做高频重复轮询。

## 6. 推荐执行顺序

建议下一阶段按下面顺序推进：

1. 运行批次沉淀与回放；
2. agent 中间产物模型；
3. `auction_wave` 卖出规则显式化；
4. 记录模型统一；
5. Hub 按批次展示；
6. 市场层与标的层建设；
7. 买入 trigger 与盘口确认；
8. 风控体系增强；
9. 真实东方财富适配器；
10. 认证、权限、告警与平台化。

## 7. 里程碑定义

为了避免后续开发一直处在“半完成”状态，建议给 Trader 定义三个里程碑：

### M1：可验证

满足条件：

- 单次运行结果可回放；
- 策略规则可解释；
- agent 中间产物可查看；
- Hub 能按批次查看；
- 不依赖重复外部请求就能完成事后分析。

### M2：可控实验 agent

满足条件：

- 市场层、标的层、触发层都已经成型；
- 买入侧和卖出侧规则都基本成型；
- 风控具备账户层约束；
- stub / paper 行为稳定；
- 具备较完整测试覆盖。

### M3：受控实盘

满足条件：

- 真实东方财富适配器接入；
- 有明确授权与审计；
- 有失败处理与告警；
- 有权限边界和回滚策略。
