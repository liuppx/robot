# Trader 能力分层

本文档描述的不是“当前已经实现了什么”，而是 `Trader` 作为一个面向实际应用、并持续朝 agent 方向演进的交易机器人，应该围绕哪些核心能力建设。

和 `baseline.md` 的区别是：

- `baseline.md` 讲当前现实；
- `capability-model.md` 讲目标主轴。

如果只用一句话概括，Trader 的目标不应停留在“卖出执行机器人”，而应逐步走向：

> 一个面向短线交易的决策机器人，负责从市场热点中筛选候选标的，结合形态与盘口寻找买点，并对持仓执行规则化退出与审计管理。

从 agent 视角看，这意味着它要逐步具备一条完整闭环：

```text
观察 -> 理解 -> 筛选 -> 触发 -> 计划 -> 执行 -> 复盘
```

后续所有能力建设，都建议判断自己是在补这条闭环中的哪一段，而不是只新增一条分散规则。

## 1. 为什么核心要放在买入、选股、盘口

如果只做卖出，机器人更多是在控制回撤和执行纪律。

但如果目标是提升效果并满足实际应用，真正决定收益上限的部分是：

1. 选股是否选到了值得参与的标的；
2. 买入是否买在了高胜率的位置；
3. 盘口是否确认了当下资金行为而不是主观想象。

所以在产品层面，Trader 的能力建设应当从“卖出规则是否清楚”，升级为：

- 能不能稳定构建候选池；
- 能不能找到可信买点；
- 能不能用盘口做执行确认；
- 能不能在持仓阶段有纪律地退出。

## 2. 总体能力结构

建议把 Trader 拆成四层能力：

1. 市场层
2. 标的层
3. 触发层
4. 执行层

这四层的关系是：

```text
市场层 -> 标的层 -> 触发层 -> 执行层
```

也就是：

- 先判断今天看什么市场方向；
- 再从方向里收敛出候选标的；
- 再判断这些标的什么时候可以买、什么时候该卖；
- 最后把决策落成订单计划、风控和审计记录。

## 3. 市场层

市场层回答的问题是：

- 今天市场主线是什么；
- 哪些板块或题材正在增强；
- 当前环境适不适合出手。

### 3.1 目标

目标不是预测整个市场，而是给机器人一个“今天重点看什么”的优先级。

### 3.2 应关注的能力

建议沉淀这些能力：

1. 板块热度排序
2. 题材持续性判断
3. 市场风格识别
4. 情绪环境判断

### 3.3 可量化的参考特征

可以逐步考虑：

- 板块涨幅中位数
- 板块涨停数 / 炸板数
- 板块成交额增量
- 龙头股相对强弱
- 热点扩散度
- 持续天数
- 连板高度分布

### 3.4 输出结果

市场层建议输出：

- `theme_rankings`
- `market_regime`
- `trade_permission`

其中 `trade_permission` 可以是很实际的结果，例如：

- 今天允许正常交易
- 今天只允许轻仓试仓
- 今天只观察不出手

## 4. 标的层

标的层回答的问题是：

- 在当前市场主线里，哪些具体标的值得进入观察池；
- 这些标的处于什么位置；
- 哪些标的虽然热，但现在不适合买。

### 4.1 目标

把全市场压缩成一个小而高质量的候选池。

### 4.2 标的层的三类能力

#### 4.2.1 候选池筛选

先从市场主线里筛出候选标的。

应考虑：

- 是否属于热点核心
- 是否有足够成交额
- 是否具备相对强势
- 是否值得盘中继续盯

#### 4.2.2 形态判断

这里才进入你关心的二浪、三浪、平台、突破等问题。

但工程上不建议一开始直接输出“这是二浪/三浪”，而应先拆成可计算特征：

- 是否刚突破前高
- 是否处于平台末端
- 是否经历过首波拉升后回踩
- 回踩幅度是否可接受
- 当前是否离主升段启动位足够近

之后再把这些特征组合成更贴近人的语言：

- 更像二浪
- 更像三浪
- 更像追高
- 更像接近末端

#### 4.2.3 压力位判断

压力位是“能不能买”的核心约束之一。

需要识别：

- 前高压力
- 密集成交区压力
- 连续拉升后的衰竭区
- 冲高后回落的历史套牢区

### 4.3 输出结果

标的层建议输出：

- `watchlist_candidates`
- `pattern_assessment`
- `resistance_assessment`
- `candidate_grade`

候选池最终建议收敛为：

- `A 级`：重点盯盘口，允许出手
- `B 级`：继续观察，不急出手
- `C 级`：排除

## 5. 触发层

触发层回答的问题是：

- 什么时候买；
- 为什么是现在买；
- 什么情况下卖；
- 什么情况下应该放弃。

这是连接“分析”和“执行”的核心一层。

### 5.1 买入触发

买入触发不应只是一个 if 条件，而应拆成多类 trigger。

建议至少区分：

1. `breakout_trigger`
2. `limit_up_attempt_trigger`
3. `sell_wall_clear_trigger`
4. `reseal_trigger`
5. `pullback_reentry_trigger`

每个 trigger 都应记录：

- 触发时间
- 触发原因
- 前置条件
- 风险提示

### 5.2 盘口触发

盘口是触发层里的高精度确认器。

盘口不应停留在模糊语言，而应拆成三类信号：

#### 5.2.1 挂单结构

- 卖一到卖五是否明显减弱
- 买一到买五是否持续抬升
- 是否存在压单
- 压单是被吃掉还是被撤掉

#### 5.2.2 成交行为

- 主动买单是否放大
- 冲板时是否被快速打穿
- 炸板后是否有承接
- 回封速度是否足够快

#### 5.2.3 封板质量

- 首次封板时间
- 封单量级
- 炸板次数
- 回封成功率
- 封板后卖压是否放大

### 5.3 卖出触发

卖出依旧重要，但在总体能力结构中，它更适合作为“持仓治理”能力存在。

当前优先应包括：

- 竞价区间卖出
- 回落阈值卖出
- 午后强制退出
- 风险事件退出

### 5.4 输出结果

触发层建议输出：

- `entry_trigger`
- `exit_trigger`
- `trigger_confidence`
- `decision_reason`

## 6. 执行层

执行层回答的问题是：

- 这一刻到底做不做；
- 做多少；
- 是否允许下单；
- 如何记录和复盘。

### 6.1 风控

至少应覆盖：

- 单票最大仓位
- 单日最大亏损
- 单日最大交易次数
- 总暴露限制
- 重复下单保护
- 外部接口异常降级

### 6.2 订单计划

实际应用里，机器人不应只有“买入”或“卖出”两个动作，而应输出更完整的订单计划：

- 试仓金额
- 确认后追加金额
- 最大允许仓位
- 撤单条件
- 失效条件

### 6.3 审计与回放

要满足实际应用，执行层必须支持：

- 每轮运行留痕
- 每条触发可解释
- 每笔订单可复盘
- 尽量减少对外部接口的重复调用

### 6.4 Hub 管理

执行层最终要和 Hub 对齐，让用户可以：

- 看候选池
- 看触发记录
- 看订单计划
- 看执行结果
- 看持仓状态
- 看失败原因

## 7. 实际应用下的推荐运行模式

在当前阶段，不建议一开始追求完全自动化。

更实际的模式是：

### 7.1 半自动模式

机器人负责：

- 市场筛选
- 候选池生成
- 买卖触发判断
- 订单计划生成
- 风控检查

人负责：

- 最终确认是否下单
- 复核高风险动作
- 处理异常情形

### 7.2 受控自动模式

只有在这些条件成熟后再推进：

- 数据稳定
- 回放机制稳定
- 触发误报率可接受
- 风控体系完整
- 审计闭环完整

## 8. 如何提升效果

Trader 效果提升不应依赖“继续堆规则”，而应依赖更清晰的层次化建设。

### 8.1 先提升候选池质量

优先级最高。

因为如果观察池本身质量差，后面的盘口和买点只是在错误标的上精细化操作。

### 8.2 再提升买点触发质量

重点是：

- 减少误触发
- 提高确认强度
- 区分试仓和确认加仓

### 8.3 最后提升盘口精度

盘口是提高执行精度的重要抓手，但前提是你有足够可靠的盘口数据。

如果当前数据源拿不到稳定盘口深度，就应先做“盘口近似版”，不要假装已经实现真实盘口策略。

## 9. 对当前 Trader 的现实影响

这份能力分层文档不是说当前代码已经具备上述能力，而是说明：

- 当前代码主轴仍偏执行和卖出；
- 未来真正要提升效果，主轴必须转向选股、买点、盘口；
- 卖出、风控、Hub、审计属于必要支撑层，但不应继续被误当成唯一核心。

## 10. 建议的下一步建设顺序

如果按这份能力模型推进，建议顺序是：

1. 建立市场层的板块/题材热度输出；
2. 建立标的层的候选池和位置评估；
3. 定义统一的买入 trigger / 卖出 trigger 模型；
4. 明确盘口数据可得性与替代方案；
5. 将触发层结果接入执行层；
6. 在 Hub 中分别展示市场、候选池、触发、执行结果。
