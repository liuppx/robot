# Trader 决策记录

本文档用于记录 `Trader` 当前已经明确的关键产品和工程决策，避免后续在同一个问题上反复摇摆。

它不替代：

- `baseline.md`
- `capability-model.md`
- `roadmap.md`

它只回答一件事：

> 当前哪些判断已经确认，可以作为后续设计和开发的约束。

## D-001 当前现实与目标方向分开描述

### 决策

`Trader` 的文档和讨论中，必须区分：

1. 当前现实
2. 目标方向

### 原因

当前代码主轴仍然偏：

- 已关注标的评估
- 卖出规则执行
- 运行审计
- Hub 展示

但产品目标已经不应只停留在“卖出执行机器人”，而应逐步走向：

- 选股
- 买点
- 盘口
- 持仓退出

如果不把“现实”和“目标”分开写，文档会要么夸大当前能力，要么低估后续方向。

### 影响

- `README.md` 同时说明当前现实和目标方向；
- `baseline.md` 只描述当前现实；
- `capability-model.md` 只描述目标能力模型。

## D-002 当前主路径仍是执行与审计，不是假装已经完成选股买点

### 决策

在代码和基线文档层面，当前 `Trader` 仍按“执行与审计主导”的现实来描述，不把未完成的选股、买点、盘口能力包装成已落地功能。

### 原因

当前真正稳定的部分是：

- `run_once` / `run-loop`
- 策略执行
- 风控检查
- `paper` / `eastmoney_stub`
- `runtime/` 记录
- Hub 工作台展示

而选股、买点、盘口，目前仍主要处于目标建模阶段。

### 影响

- 当前版本不对外宣称“完整自动选股买卖机器人”；
- 所有新能力都应区分“已实现”和“目标能力”。

## D-003 目标主轴切换到选股、买点、盘口

### 决策

Trader 的中长期核心能力建设，主轴明确放在：

1. 选股
2. 买点
3. 盘口确认
4. 持仓退出

### 原因

如果只做卖出规则，机器人更多是在控制回撤和执行纪律。

而真正决定效果和实际应用价值的，是：

- 候选池质量
- 买点质量
- 盘口确认质量

### 影响

- 后续模型设计优先考虑市场层、标的层、触发层；
- 执行层、Hub、审计作为支撑层继续演进；
- 不再把“卖出规则是否清楚”当成唯一核心问题。

## D-004 优先保存运行结果，减少同花顺重复调用

### 决策

在同花顺接口额度受限的前提下，优先沉淀运行结果和回放能力，而不是依赖重复实时调用做分析。

### 原因

当前外部接口额度有限，反复请求会直接压缩测试和迭代空间。

更合理的方式是：

- 把一次运行用到的数据保存下来；
- 让后续分析、复盘、前端展示尽量复用本地运行结果。

### 影响

- `runtime/` 产物将继续增强；
- `roadmap.md` 中“运行批次沉淀与回放”保持高优先级；
- Hub 后续应更多围绕“运行批次”而不是纯滚动摘要组织视图。

## D-005 当前不接真实东方财富实盘

### 决策

当前阶段不接真实东方财富实盘下单。

### 原因

在以下条件未满足前，接真实实盘风险过高：

- 运行结果不可回放
- 触发逻辑不稳定
- 风控体系不完整
- 审计闭环不完整

### 影响

- 当前 broker 仅使用 `paper` 或 `eastmoney_stub`；
- 所有与真实券商接入相关的工作都放在后续阶段；
- 文档中必须明确“当前不会发送真实订单”。

## D-006 当前更适合半自动，而不是全自动

### 决策

Trader 的实际应用路径优先采用“半自动决策支持”，而不是直接追求全自动短线交易。

### 原因

当前最适合机器人承担的是：

- 市场筛选
- 候选池生成
- 触发识别
- 订单计划生成
- 风控检查

而高风险动作仍适合保留人工确认。

### 影响

- 后续产品设计优先支持“建议 + 确认 + 审计”模式；
- 不以短期内实现完全自动实盘为目标。

## D-007 Trader 文档采用分层组织

### 决策

Trader 文档保持以下结构：

- `README.md`：总入口
- `baseline.md`：当前现实
- `capability-model.md`：目标能力模型
- `roadmap.md`：演进顺序
- `run_once_sequence.puml`：当前执行链路
- `runbook.md`：运行手册
- `decisions.md`：关键决策记录

### 原因

当前 Trader 同时涉及：

- 产品定位
- 工程基线
- 未来路线
- 运行机制

如果不拆开，单篇文档会混乱且容易失真。

### 影响

- 后续补文档优先进入现有分层；
- 不再继续把所有内容堆到 `README.md`。

## D-008 Trader 持续朝 agent 方向演进

### 决策

Trader 后续不按“单一策略脚本”演进，而按“交易 agent”方向持续迭代。

### 原因

如果只是一个策略脚本，它更适合回答：

- 某个条件是否触发
- 某个动作是否执行

但你希望的 Trader 更接近一个 agent，需要能持续承担：

- 市场理解
- 候选池生成
- 机会判断
- 订单计划
- 风控约束
- 审计回放
- 与 Hub 的交互管理

也就是说，Trader 不只是执行某条规则，而是逐步形成“观察 - 判断 - 计划 - 执行 - 复盘”的闭环能力。

### 影响

- `capability-model.md` 中的市场层、标的层、触发层、执行层继续保留；
- 后续设计优先考虑可组合能力，而不是继续把逻辑堆进单个策略类；
- Hub 中应逐步展示 agent 的中间产物，例如候选池、触发记录、订单计划和复盘结果；
- 文档、代码和产品讨论中，默认把 Trader 视为交易 agent 的演进起点。
