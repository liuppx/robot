# Coder Bot 使用手册

## 1. 标准链路

系统执行链路：

1. GitHub 侧通过标签 `ai-run`，或 `RUN_ON_ISSUE_OPENED=true` 发现 issue
2. 服务创建或复用 issue session，并在飞书 handoff 群里创建讨论线程
3. 先在同一个飞书线程里讨论方案
4. 在线程里发送 `/run` 或其他确认词
5. 服务才真正入队并调 OpenClaw 执行
6. 执行结束后，在同一个飞书线程回结果

执行入口说明：

- GitHub 侧负责发现 issue 并发起 handoff
- 执行确认来自飞书线程
- `/run` 必须发在绑定的飞书线程里，不是 issue 评论区，也不是别的聊天窗口

常见部署方式是只开轮询：

```env
ENABLE_POLLING=true
ENABLE_WEBHOOK=false
```

推荐在线程里确认：

- `/run`
- `开始执行`
- `确认执行`
- `可以执行`

这些词来自 `FEISHU_CONFIRM_KEYWORDS`，可以按需调整。

## 2. 触发方式

默认支持两种 GitHub 触发：

- 标签 `ai-run`
- `RUN_ON_ISSUE_OPENED=true` 时自动处理新建 issue

如果你同时开了 webhook 和轮询，建议仍以标签或新建 issue 作为 GitHub 侧触发入口。

## 3. 会话模型

每个 GitHub Issue 会固定绑定：

- 一个稳定的 `session_key`
- 一个稳定的 OpenClaw `agent_id`
- 一个稳定的 issue 工作区 `data/repos/<repo>/issues/issue-<n>/`
- 至少一个绑定的飞书线程

仓库本体会放在 issue 工作区下的 `repo/` 子目录里。
这意味着同一个 issue 重跑时会复用 OpenClaw 上下文、飞书线程和工作区。

## 4. 平时怎么查状态

健康检查：

```bash
curl -s http://127.0.0.1:9081/health
```

看某个 issue 的会话：

```bash
curl -s http://127.0.0.1:9081/issues/<owner>/<repo>/<issue_number>/session
```

这个接口会返回：

- issue 状态
- session 信息
- OpenClaw agent / workspace
- 真实仓库路径
- 已绑定的飞书线程

另外可以顺手看两条配置路径：

- `openclaw_config_path`：静态源配置，应该是 `config/openclaw.json`
- `openclaw_runtime_config_path`：运行时有效配置，应该是 `data/openclaw/runtime/openclaw.runtime.json`

按飞书线程反查：

```bash
curl -s http://127.0.0.1:9081/feishu/bindings/<chat_id>/<thread_id>
```

## 5. 绑定飞书线程

```bash
curl -sX POST http://127.0.0.1:9081/feishu/bind \
  -H 'Content-Type: application/json' \
  -d '{
    "repo_full_name": "yeying-community/robot",
    "issue_number": 18,
    "chat_id": "oc_xxx",
    "thread_id": "omt_xxx"
  }'
```

解绑：

```bash
curl -sX DELETE http://127.0.0.1:9081/feishu/bindings/oc_xxx/omt_xxx
```

如果 handoff 线程已经自动建好，通常不需要手工绑定。这个接口主要用于修复绑定关系或做联调。

## 6. OpenClaw 与 GitHub 的分工

- `issue_bot_service.py` 负责 GitHub 触发、飞书线程绑定、确认后入队、仓库准备、PR 生命周期
- OpenClaw gateway 负责飞书接入、持续会话、调用 `router/gpt-5.4` 执行
- 如果仓库里存在 `scripts/sync.sh`，外层服务优先复用它做上游同步
- PR 发布由外层服务执行 `git commit`、`git push` 并通过 GitHub API 创建 PR

配置和状态目录也要分开理解：

- `config/openclaw.json`：你手工维护的静态配置
- `data/openclaw/runtime/openclaw.runtime.json`：服务自动生成的有效配置
- `data/openclaw/state/`：OpenClaw 自己持久化 agent / 插件安装状态的地方

## 7. 常见操作

先看 gateway：

```bash
./scripts/status_gateway.sh
tail -f data/logs/openclaw-gateway.log
```

再看 bot：

```bash
curl -s http://127.0.0.1:9081/health
tail -f data/logs/gunicorn.error.log
```

重装或重启服务：

```bash
BOT_USER="$(id -un)" UV_BIN="$HOME/.local/bin/uv" ./scripts/install_systemd.sh
sudo systemctl restart openclaw-gateway coder-bot
```

## 8. 常见排查

如果 GitHub 侧根本没触发，优先检查：

- `ALLOWED_REPOS` 是否包含目标仓库
- `TRIGGER_LABEL` 或 `RUN_ON_ISSUE_OPENED` 是否按预期配置
- `/health` 里的 `last_poll_error`
- `data/state.json` 的轮询缓存

如果飞书侧建了线程，但讨论消息或 `/run` 没反应，优先检查：

- `/run` 是否发在绑定的线程里
- `FEISHU_CONFIRM_KEYWORDS` 是否包含 `/run`
- 飞书应用是否具备 `im:message.group_msg`
- `config/openclaw.json` 里的群 ID 是否正确
- `channels.feishu.groups.<group_id>.topicSessionMode` 是否还是 `enabled`
- `channels.feishu.groups.<group_id>.replyInThread` 是否还是 `enabled`

如果执行已经开始，但结果没有回到同一个线程，优先检查：

- 该 issue 的 binding 是否存在
- `root_message_id` / `thread_id` 是否正常写入
- `curl -s /issues/<owner>/<repo>/<issue_number>/session` 返回的绑定信息
- `tail -f data/logs/gunicorn.error.log`
