# 使用手册

## 1. 命令入口

所有入口都在飞书 handoff 群里。

可用命令：

- `/help`
- `/issue <repo>`
- `/issue <repo> #<issue_number>`
- `/issue <repo> #<issue_number> <model>`
- `/model <name>`
- `/model default`
- `/run`

## 2. 基本流程

1. 发送 `/issue deployer`
2. 机器人返回该仓库最近待处理 issue 列表
3. 发送 `/issue deployer #108`
4. 机器人创建或复用这个 issue 的讨论线程
5. 在线程里讨论方案
6. 发送 `/run` 开始执行
7. 等待机器人回帖进度和最终结果

## 3. 模型选择

系统是模型优先，不再单独选择执行器。

规则：

- `gpt-*` 自动走 `Codex`
- `claude-*` 自动走 `Claude`

示例：

```text
/issue deployer #108 gpt-5.5
/model gpt-5.4-mini
/model claude-opus-4-6
/model default
```

说明：

- `/model default` 会恢复到系统默认模型
- 如果切模型导致后端变化，旧会话会自动清空，避免串 session
- 已经排队的 job 会冻结当时的模型和后端，不会被后续 `/model` 污染

## 4. 执行确认

线程里这些内容都会触发执行：

- `/run`
- `开始执行`
- `确认执行`
- `可以执行`
- `方案1`
- `执行方案1`

建议直接使用 `/run`，语义最明确。

## 5. 结果怎么看

线程里的回帖通常分三类：

- 讨论回复
- 执行进度
- 最终结果

如果成功，最终结果里会包含：

- 实际执行模型
- 结果摘要
- PR 链接

## 6. 常见使用示例

查看仓库待处理 issue：

```text
/issue robot
```

直接进入指定 issue 并选模型：

```text
/issue deployer #108 claude-opus-4-6
```

在线程里切回默认模型：

```text
/model default
```

开始执行：

```text
/run
```

## 7. 常见问题

### 7.1 为什么 `/issue deployer #108 codex gpt-5.5` 被拒绝

因为现在不再支持单独指定执行器。

正确写法：

```text
/issue deployer #108 gpt-5.5
```

### 7.2 为什么我改了 `/model`，但已经排队的任务没变

因为 job 在入队时就冻结了执行规格。你改的是后续任务的默认值，不会反向改写已经排队的那次执行。

### 7.3 为什么线程里只看到“已启动，正在准备仓库”

这通常表示任务还没跑到模型执行，常见原因是：

- 拉仓库超时
- repo lock 被占用
- GitHub token 或 SSH 链路异常

### 7.4 怎么确认服务当前状态

可以让运维侧检查：

```bash
curl -s http://127.0.0.1:9081/health
./scripts/status_bot.sh
tail -f data/logs/gunicorn.error.log
```
