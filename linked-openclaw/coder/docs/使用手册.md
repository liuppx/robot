# Coder Bot 使用手册

## 1. 标准链路

系统执行链路：

1. 通过issuer机器人侧在相应的 issue 评论区发送 `/run`
2. 服务创建或复用 issue session，并在飞书 handoff 群里创建讨论线程
3. 先在同一个飞书线程里讨论方案
4. 在线程里发送 `/run` 或其他确认词
5. 服务才真正入队并调 OpenClaw 执行
6. 执行结束后，在同一个飞书线程回结果

最容易混淆的点只有一个：

- GitHub issue 评论区的 `/run` = 发起 handoff
- 飞书讨论线程里的 `/run` = 真正开始执行

如果你只做了第一个 `/run`，但没有在线程里确认，任务不会真的开始。

## 2. 用户怎么触发

最标准的使用姿势：

1. 通过issuer机器人在对应的GitHub issue 评论区单独发一条：

```text
/run
```

3. 等机器人把这个 issue 转到飞书 handoff 群的同一条讨论线程里
4. 先在线程里讨论方案、边界和预期结果
5. 确认可以开工后，再在线程里单独发一条：

```text
/run
```

6. 等待机器人在线程里回执“已开始处理”或“执行失败”

失败后重跑规则：

- 如果某次执行失败，直接在同一个飞书线程里再次发送 `/run` 即可重跑
- 不需要手工重置 session，也不需要重新绑定线程
- 通常也不需要再回 GitHub issue 评论区重新发 `/run`

执行入口说明：

- 顶层飞书群负责发现 issue 并发起 handoff
- 顶层飞书群里的 `/issue_<repo>` 和 `/issue_<repo> <issue_number>` 负责发起 handoff
- 执行确认来自飞书线程
- 飞书线程里的 `/run` 才是真正开始执行

常见配置是只保留飞书触发：

```env
FEISHU_CONFIRM_KEYWORDS=/run,开始执行,确认执行,可以执行
DISPATCH_INTERVAL_SECONDS=5
```

推荐在线程里确认：

- `/run`
- `开始执行`
- `确认执行`
- `可以执行`

这些词来自 `FEISHU_CONFIRM_KEYWORDS`，可以按需调整。

## 3. 启动方式

当前系统采用标准包结构：

- `src/main.py`：启动入口与组件组装
- `src/webhook_server.py`：HTTP 路由
- `src/issue_service.py`：issue session / Feishu handoff / 结果回帖
- `src/worker.py`：实际执行、提交分支、创建 PR
- `src/scheduler.py`：飞书扫描与调度

手动启动：

```bash
cd "$APP_DIR"
uv sync --frozen
./scripts/start_gateway.sh
CODER_BOT_ENV_FILE=config/coder-bot.env uv run --frozen gunicorn -c config/gunicorn.conf.py src.main:APP
```

或者：

```bash
cd "$APP_DIR"
uv sync --frozen
python -m src --env-file config/coder-bot.env serve
```

systemd 方式：

```bash
cd "$APP_DIR"
BOT_USER="$(id -un)" UV_BIN="$HOME/.local/bin/uv" ./scripts/install_systemd.sh
sudo systemctl restart openclaw-gateway coder-bot
```

确认 `coder-bot` 的服务入口：

```text
src.main:APP
```

## 5. 会话模型

每个 GitHub Issue 会固定绑定：

- 一个稳定的 `session_key`
- 一个稳定的 OpenClaw `agent_id`
- 一个稳定的 issue 工作区 `data/repos/<repo>/issues/issue-<n>/`
- 至少一个绑定的飞书线程

仓库本体会放在 issue 工作区下的 `repo/` 子目录里。
这意味着同一个 issue 重跑时会复用 OpenClaw 上下文、飞书线程和工作区。

## 6. 平时怎么查状态

健康检查：

```bash
curl -s http://127.0.0.1:9081/health
```

看某个 issue 的会话：

```bash
curl -s http://127.0.0.1:9081/issues/<owner>/<repo>/<issue_number>/session
```

这个接口会返回：

- issue 状态
- session 信息
- OpenClaw agent / workspace
- 真实仓库路径
- 已绑定的飞书线程

另外可以顺手看两条配置路径：

- `openclaw_config_path`：静态源配置，应该是 `config/openclaw.json`
- `openclaw_runtime_config_path`：运行时有效配置，应该是 `data/openclaw/runtime/openclaw.runtime.json`

按飞书线程反查：

```bash
curl -s http://127.0.0.1:9081/feishu/bindings/<chat_id>/<thread_id>
```

## 7. 绑定飞书线程

```bash
curl -sX POST http://127.0.0.1:9081/feishu/bind \
  -H 'Content-Type: application/json' \
  -d '{
    "repo_full_name": "yeying-community/robot",
    "issue_number": 18,
    "chat_id": "oc_xxx",
    "thread_id": "omt_xxx"
  }'
```

解绑：

```bash
curl -sX DELETE http://127.0.0.1:9081/feishu/bindings/oc_xxx/omt_xxx
```

如果 handoff 线程已经自动建好，通常不需要手工绑定。这个接口主要用于修复绑定关系或做联调。

## 8. OpenClaw 与 GitHub 的分工

- `src/main.py` 只负责启动入口、配置加载和组件组装
- `src/webhook_server.py` 负责 health、session/binding API
- `src/issue_service.py` 负责飞书触发的 handoff、session 状态、飞书讨论与结果回帖
- `src/worker.py` 负责真正执行任务、提交分支和创建 PR
- `src/scheduler.py` 负责扫描飞书确认、分发队列任务
- OpenClaw gateway 负责飞书接入；实际 issue 讨论与执行由 Codex 驱动
- 如果仓库里存在 `scripts/sync.sh`，外层服务优先复用它做上游同步
- PR 发布由外层服务执行 `git commit`、`git push` 并通过 GitHub API 创建 PR

配置和状态目录也要分开理解：

- `config/openclaw.json`：你手工维护的静态配置
- `data/openclaw/runtime/openclaw.runtime.json`：服务自动生成的有效配置
- `data/openclaw/state/`：OpenClaw 自己持久化 agent / 插件安装状态的地方

## 9. 常见操作

先看 gateway：

```bash
./scripts/status_gateway.sh
sudo journalctl -u openclaw-gateway -f
```

再看 bot：

```bash
curl -s http://127.0.0.1:9081/health
tail -f data/logs/gunicorn.error.log
```

重装或重启服务：

```bash
BOT_USER="$(id -un)" UV_BIN="$HOME/.local/bin/uv" ./scripts/install_systemd.sh
sudo systemctl restart openclaw-gateway coder-bot
sudo systemctl status --no-pager coder-bot
```

检查 `coder-bot` 当前是不是新入口：

```bash
sudo systemctl status --no-pager coder-bot
```

你应该看到：

```text
... gunicorn -c /your/app/config/gunicorn.conf.py src.main:APP
```

## 10. 执行结果怎么看

成功时，机器人会在同一个飞书线程里回：

- 已完成
- 对应的 PR 链接
- 简短结果摘要

失败时，机器人会在同一个飞书线程里回：

- 执行失败
- issue 编号 / job id
- 错误摘要

这时优先做两件事：

1. 先看线程里返回的错误摘要
2. 如果只是临时失败，直接在同一个线程里再次发送 `/run`

## 11. 常见排查

如果顶层飞书群里 `/issue_*` 没触发，优先检查：

- `ALLOWED_REPOS` 是否包含目标仓库
- `REPO_ALIASES` 是否补齐了仓库别名
- `/health` 里的 `allowed_repos`、`queue`

如果飞书侧建了线程，但讨论消息或 `/run` 没反应，优先检查：

- `/run` 是否发在绑定的线程里
- `FEISHU_CONFIRM_KEYWORDS` 是否包含 `/run`
- 飞书应用是否具备 `im:message.group_msg`
- `config/openclaw.json` 里的群 ID 是否正确
- `channels.feishu.groups.<group_id>.topicSessionMode` 是否还是 `enabled`
- `channels.feishu.groups.<group_id>.replyInThread` 是否还是 `enabled`

如果执行已经开始，但结果没有回到同一个线程，优先检查：

- 该 issue 的 binding 是否存在
- `root_message_id` / `thread_id` 是否正常写入
- `curl -s /issues/<owner>/<repo>/<issue_number>/session` 返回的绑定信息
- `tail -f data/logs/gunicorn.error.log`

如果任务行为和当前代码不一致，优先检查：

- `sudo systemctl status --no-pager coder-bot` 里的启动命令是否为 `src.main:APP`
- 是否已经重新执行过 `./scripts/install_systemd.sh`
- 是否已经 `sudo systemctl restart openclaw-gateway coder-bot`
