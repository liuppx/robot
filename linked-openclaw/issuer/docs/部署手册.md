# Issuer 部署手册

## 1. 适用范围

这份手册面向一台 Linux 机器，从零部署当前目录下这套 `Issuer + OpenClaw Gateway`。

系统分工：

- Feishu 侧负责接收用户请求和返回草案 / 执行结果
- OpenClaw 负责会话、模型调用、hook 装载和技能执行
- GitHub 工具负责真正的 issue create / update / close / comment / assignees 追加
- `confirmation-bridge` 负责 `/show`、`/show all`、`/edit`、`/confirm`、`/cancel`、`/issue`、`/close`、`/assignees`
- 如果消息里有图片或文件附件，附件会在执行阶段上传到外部存储，再回写到 GitHub issue / comment 正文

部署目录不是固定的。下面统一用 `APP_DIR` 表示你的实际部署目录。

例如：

```bash
export APP_DIR=/opt/deploy/issuer
```

默认端口：

- `issuer` 本地 gateway：`18890`

## 2. 机器前置条件

建议环境：

- Ubuntu 22.04 / 24.04
- 能使用 `sudo`
- 已安装 `systemd`
- 机器可以访问：
  - `api.github.com`
  - `https://open.feishu.cn`
  - `https://router.yeying.pub`
  - 你的附件存储服务地址（如果希望飞书图片真正出现在 GitHub 正文里）

先装基础工具：

```bash
sudo apt update
sudo apt install -y \
  git curl jq ca-certificates unzip \
  openssh-client build-essential \
  python3
```

## 3. 安装 `openclaw`

本项目使用仓库内维护的 OpenClaw 配置：

- `config/openclaw.json`
- `data/openclaw/state/`

安装 CLI 后，直接使用仓库内配置即可。

推荐方式一：官方安装脚本

```bash
curl -fsSL https://openclaw.ai/install.sh | bash -s -- --no-onboard
openclaw --version
```

推荐方式二：如果你已经自己管理 Node/npm

```bash
npm install -g openclaw@latest
openclaw --version
```

注意：

- 本仓库最终调用的是 `scripts/openclaw-local`
- OpenClaw CLI 安装完成后，无需额外执行 `openclaw onboard`

## 4. GitHub 侧准备

### 4.1 GitHub App

需要准备一个 GitHub App，并安装到目标仓库。

这个 App 至少要能完成这些动作：

- 读取 issue 与标签
- 创建 issue
- 更新 issue
- 关闭 issue
- 在 issue 下发评论

### 4.2 GitHub App 私钥

把 GitHub App 下载到的私钥放到：

```bash
mkdir -p "$APP_DIR/secrets"
chmod 700 "$APP_DIR/secrets"
```

目标文件：

```bash
$APP_DIR/secrets/github-app.pem
```

建议权限：

```bash
chmod 600 "$APP_DIR/secrets/github-app.pem"
```

### 4.3 附件存储（可选但推荐）

如果你希望飞书消息里的图片在 GitHub issue / comment 正文中直接显示，需要准备一个可写的外部附件存储。

当前默认支持 WebDAV，至少需要：

- 可写的 WebDAV 目录
- 一个用于写入文件的账号
- 一个用于生成公开分享链接的接口和 bearer token

如果不配置这部分：

- 普通 issue create / update / close / comment 仍然可以工作
- 但飞书图片不会自动变成 GitHub 正文里的可预览图片

## 5. Feishu / OpenClaw 侧准备

### 5.1 Feishu 应用

你需要一个能被 OpenClaw gateway 接入的飞书应用，并拿到：

- `appId`
- `appSecret`
- 目标群 `chat_id`

至少确认：

- 应用已被拉进目标群
- 事件订阅走长连接
- 应用具备群消息接收权限

### 5.2 OpenClaw 静态配置

复制模板：

```bash
cd "$APP_DIR"
cp config/openclaw.json.template config/openclaw.json
```

至少替换这些值：

- `models.providers.router.apiKey`
- `channels.feishu.appId`
- `channels.feishu.appSecret`
- `channels.feishu.groupAllowFrom`
- `gateway.auth.token`

这条很关键：

- `messages.groupChat.visibleReplies` 建议显式设为 `automatic`
- `/edit <id> <要求>` 会直接读取 `models.providers.router.baseUrl/apiKey` 并调用上游 `/responses` 接口；这两项缺失或配错时，草案改写会失败

否则群里可能出现“处理了但没有可见回复”。

## 6. 拉代码与初始化配置

```bash
mkdir -p "$(dirname "$APP_DIR")"
git clone <your-repo-url> "$APP_DIR"
cd "$APP_DIR"
```

建议先建目录：

```bash
mkdir -p data/logs data/openclaw/state data/openclaw/workspace-larkbot secrets
```

## 7. 填写配置

### 7.1 GitHub App 配置

编辑：

```bash
vim "$APP_DIR/config/github-app.config.env"
```

最关键的字段：

```env
GITHUB_APP_ID=
GITHUB_APP_PRIVATE_KEY_PATH=secrets/github-app.pem
GITHUB_APP_INSTALLATION_ID=
GITHUB_DEFAULT_OWNER=yeying-community
GITHUB_DEFAULT_REPO=robot
WEBDAV_BASE_URL=
WEBDAV_USERNAME=
WEBDAV_PASSWORD=
WEBDAV_PUBLIC_SHARE_API_URL=
WEBDAV_PUBLIC_SHARE_BEARER_TOKEN=
```

说明：

- `GITHUB_APP_INSTALLATION_ID` 可以留空，工具会按 owner/repo 自动查询
- `GITHUB_APP_PRIVATE_KEY_PATH` 是相对 `APP_DIR` 解析的
- 如果你直接想用 token，也可以写 `GITHUB_TOKEN=...`
- 如果你的附件服务使用 `Key ID / Key Secret` 命名，也可以改用 `WEBDAV_KEY_ID` / `WEBDAV_KEY_SECRET`
- `WEBDAV_BASE_URL` 建议直接指向最终上传目录，例如 `https://webdav.example.test/dav/personal/issue-pictures`
- `WEBDAV_PUBLIC_SHARE_API_URL` 需要能把上传后的文件路径转换成公开可访问链接，否则 GitHub 正文里只能保留内部 WebDAV 地址

### 7.2 OpenClaw 配置

编辑：

```bash
vim "$APP_DIR/config/openclaw.json"
```

至少检查这些字段：

- `agents.defaults.workspace`
- `models.providers.router.apiKey`
- `channels.feishu.appId`
- `channels.feishu.appSecret`
- `channels.feishu.groupAllowFrom`
- `gateway.port`
- `gateway.auth.token`
- `messages.groupChat.visibleReplies`

推荐配置：

- `agents.defaults.workspace=$APP_DIR/data/openclaw/workspace-larkbot`
- `gateway.port=18890`
- `messages.groupChat.visibleReplies=automatic`

### 7.3 策略配置

编辑：

```bash
vim "$APP_DIR/config/policy.json"
```

这几个字段最关键：

```json
{
  "admins": [
    "ou_xxx_replace_with_admin_user_id"
  ],
  "repoAliases": [
    {
      "alias": "robot",
      "owner": "yeying-community",
      "repo": "robot"
    }
  ],
  "githubUserAliases": [
    {
      "alias": "刘鑫",
      "login": "kobofare"
    },
    {
      "alias": "张博",
      "login": "dlwlrma-333"
    }
  ]
}
```

说明：

- `repoAliases` 决定群里说 `robot`、`router` 这类简称时实际落到哪个 `owner/repo`
- `githubUserAliases` 决定正文里的 `跟进人：姓名` 能否自动同步成 GitHub assignee
- `/edit` 如果把跟进人从 A 改成 B，而没有显式指定 assignees，会按这里的映射重新计算 assignee

## 8. 首次自检与一键安装

推荐直接跑：

```bash
cd "$APP_DIR"
./scripts/bootstrap.sh
```

`bootstrap.sh` 会做这些事：

- 创建 `config/`、`secrets/`、`data/`
- 生成缺失的配置文件
- 同步 `workspace_assets/` 到运行时 workspace

首次启动：

```bash
cd "$APP_DIR"
./scripts/start_gateway.sh
./scripts/status_gateway.sh
```

建议启动后立刻做一轮验收：

1. 在飞书群里 `@issuer /help`，确认机器人有可见回复。
2. 发一条创建 issue 请求，执行 `/show <id>`、`/edit <id> ...`、`/confirm <id>`，确认草案查看、改写和 GitHub 提交都正常。
3. 发 `/show all`，确认当前群待处理草案列表正常。
4. 发 `/issue robot 5`，确认直接查询命令能返回。
5. 选一条测试 issue 执行 `/assignees robot #123 刘鑫`，确认追加 assignee 正常。
6. 再发一条带图片的创建或更新请求并 `/confirm`，确认图片会出现在 GitHub issue 正文中。

当前正文默认规则：

- 新建 issue 时，正文会自动补一行 `跟进人：`
- 如果用户创建时已经说明谁来跟进，会直接写成 `跟进人：姓名`
- 如果后续用户再指定跟进人，更新流程会回填这一行

## 9. systemd 部署与重启

安装服务：

```bash
cd "$APP_DIR"
BOT_USER="$(id -un)" ./scripts/install_systemd.sh
```

重启并检查：

```bash
sudo systemctl restart issuer-openclaw-gateway
sudo systemctl status --no-pager issuer-openclaw-gateway
journalctl -u issuer-openclaw-gateway -n 100 --no-pager
```

## 10. 运行时目录说明

- `config/`：你手工维护的静态配置
- `workspace_assets/`：源码态的 tools / hooks / skills
- `data/openclaw/workspace-larkbot/`：运行时真正执行的 workspace
- `data/openclaw/state/`：OpenClaw 自己的状态目录
- `data/openclaw/workspace-larkbot/state/pending-actions.sqlite3`：草案数据库
- `data/logs/openclaw-gateway.log`：gateway 日志
- `data/logs/issuer-audit.jsonl`：issuer 审计日志

如果你改的是 `workspace_assets/`，一定记得：

```bash
./scripts/sync_workspace.sh
./scripts/stop_gateway.sh
./scripts/start_gateway.sh
```

## 11. 常见排查

如果飞书里完全没反应，优先检查：

- `config/openclaw.json` 里的飞书 `appId` / `appSecret`
- `groupAllowFrom` 是否包含当前群的 `chat_id`
- 当前消息是否真的 `@issuer`
- `./scripts/status_gateway.sh` 的最近日志

如果有处理但群里没回消息，优先检查：

- `config/openclaw.json` 里的 `messages.groupChat.visibleReplies`
- 当前是否显式用了 `automatic`
- 日志里是否有 `dispatch complete (... replies=0)`

如果 `/show all`、`/assignees`、`/close` 这种命令“没反应”，优先检查：

- `data/logs/issuer-audit.jsonl` 里是否有 `hook.show_all_drafts`、`hook.assignees_direct`、`hook.close_direct`
- 是否出现 `hook.reply.sent`、`hook.reply.failed`、`hook.reply.skipped`
- 最近改过 `workspace_assets/` 后是否执行过 `./scripts/sync_workspace.sh` 并重启 gateway`

如果整体反应很慢，优先检查：

- 日志里是否有 `llm-idle-timeout`
- 日志里是否有 `Concurrency limit exceeded for account`
- 日志里是否有 `internal_server_error`
- 当前模型是否仍是 `router/gpt-5.4`

如果 GitHub 返回 401 或 403，优先检查：

- GitHub App 是否真的安装到了目标仓库
- `Issues` 权限是否为 `Read and write`
- `config/github-app.config.env` 里的 owner/repo 是否填对
- `secrets/github-app.pem` 是否与这个 App 对应
- 如果手填了 `GITHUB_APP_INSTALLATION_ID`，它是否属于同一个安装

如果 `/edit` 失败，优先检查：

- `config/openclaw.json` 里的 `models.providers.router.baseUrl` / `apiKey`
- router 上游是否可访问，额度是否正常
- `issuer-audit.jsonl` 里是否有 `hook.edit_draft` 和 `draft.rewrite.failed`

如果跟进人没有同步成 GitHub assignee，优先检查：

- `config/policy.json` 的 `githubUserAliases` 是否配置了这个人
- alias 和飞书里输入的姓名是否能严格匹配
- `/edit` 是否显式把 `assignees` 改成了其它值

如果 issue 能创建但飞书图片没有出现在 GitHub 正文里，优先检查：

- `config/github-app.config.env` 里的 `WEBDAV_BASE_URL`、`WEBDAV_USERNAME`、`WEBDAV_PASSWORD`
- `WEBDAV_PUBLIC_SHARE_API_URL` 和 `WEBDAV_PUBLIC_SHARE_BEARER_TOKEN` 是否可用
- 机器是否能访问你的 WebDAV 服务和公开分享接口
- 修改 `workspace_assets/` 后是否执行过 `./scripts/sync_workspace.sh` 并重启 gateway
