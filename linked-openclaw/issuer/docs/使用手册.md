# Issuer 使用手册

## 1. 当前支持什么

当前主要分两类能力：

1. 草案型操作：
   - 创建 issue
   - 更新 issue
2. 直接命令：
   - `/issue <repo> [limit]` 列出最近 open issues
   - `/close <repo> #<number>` 直接关闭 issue
   - `/assignees <repo> #<number> <who>` 直接追加 assignees

补充说明：

- 显式 issue 评论请求会直接执行，不走草案。
- `/edit <id> <要求>` 只改写 pending 草案，不会直接写 GitHub。
- `/show all` 只列出当前群里状态仍为 `pending` 的草案。

## 2. 推荐使用链路

群里最顺的方式是这样：

1. `@issuer` 提出创建或更新 issue 的需求，消息里明确仓库名。
2. 系统生成草案，返回 4 位短草案 ID。
3. 用 `/show <id>` 检查当前标题、正文、标签、指派、跟进人。
4. 如果要整份改写，发 `/edit <id> <修改要求>`。
5. 如果只是补几句正文，可发 `<id> 补充：...`。
6. 确认无误后发 `/confirm <id>`。
7. 不需要了就发 `/cancel <id>`。

建议：

- 在群里尽量始终使用显式草案 ID。
- 不要再用旧的 `draft:xxxx` 写法。
- 多草案并存时，先 `/show all`，再选 ID 继续操作。

## 3. 草案协作规则

草案的创建隔离维度仍然是：

- 同一个群会话
- 同一个发起人
- 同一个仓库

这意味着：

- 同一个人对同一个仓库重复提需求时，新草案会覆盖旧草案。
- 同一个人对不同仓库可以同时挂多份草案。
- 不同人创建的草案互不覆盖。

但草案操作已经是群协作模式：

- 同群成员都可以 `/show <id>`
- 同群成员都可以 `/edit <id> <要求>`
- 同群成员都可以 `/confirm <id>`
- 同群成员都可以 `/cancel <id>`

所以要把“谁创建的草案”和“谁能在群里继续处理草案”分开理解。

## 4. 常见命令示例

- 创建：`@issuer 在 robot 创建 issue，标题是“登录接口超时”，内容是“补重试和告警”`
- 更新：`@issuer 修改 robot #123，把标题改成“登录接口偶发超时”，正文补充复现步骤`
- 评论：`@issuer 给 robot #123 评论：“请测试同学今天回归验证”`
- 查看单个草案：`/show abcd`
- 查看当前群全部草案：`/show all`
- 整份改写草案：`/edit abcd 跟进人改成刘鑫，正文补上复现步骤和期望结果`
- 补充草案正文：`abcd 补充：增加期望结果和风险说明`
- 提交草案：`/confirm abcd`
- 取消草案：`/cancel abcd`
- 查看 issue：`/issue robot`
- 查看最近 10 条 issue：`/issue robot 10`
- 直接关闭：`/close robot #123`
- 追加指派人：`/assignees robot #123 刘鑫`

## 5. 跟进人与 assignee 的规则

系统对“跟进人”和 GitHub `assignee` 的处理是分层的：

1. 创建或更新草案时，正文里会保留 `跟进人：姓名` 这一行。
2. 如果 `config/policy.json` 配了 `githubUserAliases`，系统会把跟进人自动映射成 GitHub assignee。
3. 如果 `/edit` 改了 `followOwner`，而没有显式指定 `assignees`，系统会按新的跟进人重新计算 assignee。
4. 如果没有配映射，正文里仍会保留 `跟进人：姓名`，但 GitHub assignee 可能为空。

一个典型的 `policy.json` 片段：

```json
{
  "githubUserAliases": [
    { "alias": "刘鑫", "login": "kobofare" },
    { "alias": "张博", "login": "dlwlrma-333" }
  ]
}
```

## 6. 当前启动方式

手动启动：

```bash
cd "$APP_DIR"
./scripts/start_gateway.sh
./scripts/status_gateway.sh
```

如果你改了 `workspace_assets/`：

```bash
cd "$APP_DIR"
./scripts/sync_workspace.sh
./scripts/stop_gateway.sh
./scripts/start_gateway.sh
```

systemd 方式：

```bash
cd "$APP_DIR"
BOT_USER="$(id -un)" ./scripts/install_systemd.sh
sudo systemctl restart issuer-openclaw-gateway
```

内部入口说明：

- `scripts/start_gateway.sh`：给人手工执行
- `scripts/run_gateway.sh`：给后台进程和 systemd 执行
- `scripts/openclaw-local`：实例级 OpenClaw wrapper

帮助入口：

```text
@issuer /help
```

## 7. 平时怎么查状态

先看 gateway：

```bash
./scripts/status_gateway.sh
tail -f data/logs/openclaw-gateway.log
```

再看 issuer 审计：

```bash
tail -f data/logs/issuer-audit.jsonl
```

看当前 pending：

```bash
./scripts/inspect_pending.sh summary
./scripts/inspect_pending.sh list
```

按群查：

```bash
./scripts/inspect_pending.sh conversation --conversation-id chat:oc_xxx
```

按用户查：

```bash
./scripts/inspect_pending.sh requester --requester-id ou_xxx
```

按 draftId 反查详情：

```bash
./scripts/inspect_pending.sh show --draft-id 5d496c27
```

几个特别有用的观察点：

- `data/logs/issuer-audit.jsonl`：最适合看 `/show all`、`/edit`、`/close`、`/assignees` 这类命令到底有没有命中 hook。
- `data/openclaw/workspace-larkbot/state/pending-actions.sqlite3`：草案真实状态。
- `./scripts/inspect_pending.sh list`：当前待处理草案总览。

## 8. OpenClaw 与 GitHub 的分工

- `workspace_assets/tools/github_app_token.mjs` 负责 GitHub App 认证
- `workspace_assets/tools/github_issue_create.mjs` 负责 create preview / execute
- `workspace_assets/tools/github_issue_update.mjs` 负责 update preview / execute
- `workspace_assets/tools/github_issue_close.mjs` 负责 close preview / execute
- `workspace_assets/tools/github_issue_comment.mjs` 负责明确 issue comment 的直接执行
- `workspace_assets/tools/pending_action.mjs` 负责草案保存、读取、执行、清理
- `workspace_assets/tools/draft_rewrite.mjs` 负责 `/edit <id> <要求>` 的整份改写
- `workspace_assets/hooks/confirmation-bridge/handler.ts` 负责 `/help`、`/show`、`/edit`、`/confirm`、`/cancel`、`/issue`、`/close`、`/assignees`
- OpenClaw gateway 负责 Feishu 接入、会话、模型调用和 hook 装载

配置和状态目录也要分开理解：

- `config/openclaw.json`：你手工维护的静态配置
- `config/github-app.config.env`：GitHub App / token 配置
- `data/openclaw/workspace-larkbot/`：运行时真正执行的 workspace
- `data/openclaw/state/`：OpenClaw 自己的状态目录

## 10. 常见排查

如果飞书里完全没反应，优先检查：

- `config/openclaw.json` 里的 `channels.feishu.appId` / `appSecret`
- `channels.feishu.groupAllowFrom` 是否包含当前群 `chat_id`
- 当前消息是否真的 `@issuer`
- `./scripts/status_gateway.sh` 的最近日志

如果有处理但群里没回消息，优先检查：

- `config/openclaw.json` 里的 `messages.groupChat.visibleReplies`
- 当前是否显式用了 `automatic`
- 日志里是否出现 `dispatch complete (... replies=0)`

如果整体反应很慢，优先检查：

- 上游模型是否超时
- 日志里是否有 `llm-idle-timeout`
- 日志里是否有 `Concurrency limit exceeded for account`
- 日志里是否有 `internal_server_error`、`unexpected EOF`

如果 `/show all`、`/assignees`、`/close` 这种命令“没反应”，优先检查：

- `data/logs/issuer-audit.jsonl` 里有没有对应的 `hook.show_all_drafts`、`hook.assignees_direct`、`hook.close_direct`
- 有没有 `hook.reply.sent`、`hook.reply.failed`、`hook.reply.skipped`
- 最近是否改过 `workspace_assets/` 但没执行 `./scripts/sync_workspace.sh`
- gateway 是否其实还跑着旧代码

如果 `/edit` 失败，优先检查：

- `config/openclaw.json` 里的 `models.providers.router.baseUrl` / `apiKey`
- router 配额或上游 `/responses` 接口是否正常
- `issuer-audit.jsonl` 里是否有 `hook.edit_draft` 和 `draft.rewrite.failed`

如果 `/confirm` 结果不对，优先检查：

- 当前会话里是否确实存在 pending action
- `./scripts/inspect_pending.sh show --draft-id ...` 看到的 `title/body/labels/assignees/followOwner` 是否就是你预期提交的值
- `issuer-audit.jsonl` 里是否有 `hook.confirm.executed` 或 `hook.confirm.execute_failed`

如果跟进人没有同步成 GitHub assignee，优先检查：

- `config/policy.json` 里有没有对应的 `githubUserAliases`
- 跟进人文本是否和 alias 真能匹配上
- `/edit` 是否显式把 `assignees` 改成了别的值

如果你改了源码但效果没变，优先检查：

- 是否已经执行 `./scripts/sync_workspace.sh`
- 是否已经重启 gateway
- 你改的是 `workspace_assets/` 还是 `data/openclaw/workspace-larkbot/`

## 11. 配置读取说明

`github_app.mjs` 不是自动优先读系统全局环境，而是按这个顺序读：

1. `GITHUB_APP_ENV_FILE` 或 `GITHUB_ENV_FILE`
2. 从运行时 workspace 向上找应用根目录
3. 默认读取 `APP_DIR/config/github-app.config.env`

所以当前这套实现是目录内配置，不再依赖旧的用户级 OpenClaw 配置目录。
